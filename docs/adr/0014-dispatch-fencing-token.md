# ADR 0014: Dispatch Fencing Token on Claim/Complete

**Status:** Accepted  
**Deciders:** srallapally  
**Date:** 2026-07-18

---

## Context

`job_runs.dispatch_id` existed as a column and was generated by `RunControlService.retryRun`/`redriveRun`, but was never minted at normal claim time and never read back anywhere. `RunStore.claimRun`, `markSucceeded`, `markFailed`, and `recordRuntimeExecution` all guarded state transitions only on `state`, never on `dispatch_id`. This is tracked as COR-1.

This allowed a real race: the stale-run sweeper (or, before SEC-3, a forged completion call) flips a still-alive `RUNNING` run to `FAILED` → an operator retries it (back to `QUEUED`) → it's re-claimed (`RUNNING` again, a distinct attempt) → the *original* ghost subprocess from the first attempt eventually finishes and calls `markSucceeded`/`markFailed`, which — guarded only on `state='RUNNING'` — succeeds unconditionally against the row now on the second attempt, silently overwriting or corrupting its outcome.

---

## Decision

`RunStore.claimRun` now mints a fresh `dispatch_id` (`randomUUID()`) on *every* successful claim — not just retry/redrive — and returns it. This value uniquely identifies "the currently-claimed attempt," re-minted each time a run transitions `QUEUED → RUNNING`, regardless of how it reached `QUEUED`.

`recordRuntimeExecution`, `markSucceeded`, and `markFailed` all accept an optional `dispatchId`. When provided, the UPDATE is fenced with `AND dispatch_id = $N`, so a caller holding a stale token from an earlier attempt cannot affect the current one. When omitted, they behave exactly as before (state-only guard) — this keeps the change backward compatible for the one caller intentionally left alone (see below) rather than introducing a hard requirement that could silently no-op on an overlooked call site.

The token is threaded through the full production dispatch path:
- `WorkerRunService.claimQueuedRun` attaches the minted `dispatchId` onto the claimed run object.
- `dispatchRun` (isolated/production mode) passes it to `WorkerServiceRuntimeLauncher.launchExecution`, which forwards it in the `/execute` POST body (a sibling field, not part of the job-facing runtime context — it's fencing metadata, not something job code should see) — and to `recordRuntimeExecution`/`markFailed`.
- `workerApp.js`'s `/execute` handler reads it back off the request and threads it into the `onExecutionSuccess`/`onExecutionError` callbacks.
- `src/workers/app.js`'s callbacks pass it to `runStore.markSucceeded`/`markFailed`.
- `executeRunLocallyInternal` (local-execution mode, used by local dev) passes the claimed run's `dispatchId` to its own `markSucceeded`/`markFailed` calls.
- `StaleRunSweeper._sweepRunning` now reads `dispatch_id` alongside `run_id` from `listStaleRunningIds` and passes it through to its own `markFailed` call.

`WorkerRunService.completeRun` (dead code since SEC-3 removed its route, kept only for its existing unit tests) is deliberately **not** threaded with a `dispatchId` — it has no production caller to source one from, and forcing the fencing guard there would just break its own tests for no operational benefit.

---

## Consequences

### What this closes

A ghost subprocess from a superseded attempt can no longer complete/clobber the run's current attempt. Verified end-to-end against a real Postgres instance (not just mocked): claim → sweeper-fail → retry → re-claim → ghost's stale-token completion is rejected; the real, current attempt's completion still succeeds.

### What does not change

- `RunControlService.retryRun`/`redriveRun`'s own `dispatch_id` generation — left as-is; it's superseded by the fresh mint at the next claim, but harmless as a QUEUED-state placeholder in the meantime.
- `LocalRunStore` (SQLite, local dev) — not fenced; `dispatchId` is simply ignored where destructured, matching its established pattern of holding local dev to a simpler bar than production (same precedent as SEC-1's local fallback, COR-7's and COR-3's scoping to the ES-backed service).
- COR-2 (real cancellation) — independent, separate ticket; this ADR's fencing infrastructure (the per-claim `dispatchId`) is exactly what COR-2's real-cancellation work will reuse as the identifier for "which attempt to cancel," instead of the never-populated `executionId` field.
